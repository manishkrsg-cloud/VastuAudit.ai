## Coding conventions

### Python
- Type hints on **every** function. Pydantic v2 for all DTOs.
- `async`/`await` for all I/O. No sync DB calls inside request handlers.
- `uv` for deps; `ruff` + `black`; `pytest` + `pytest-asyncio`. Target >80 % coverage on `src/core/`.
- SQLAlchemy 2.0 style with `Mapped[]` and `mapped_column`. UUID primary keys.
- Settings live in `src/config.py` (pydantic-settings). Never read `os.environ` from business logic.
- Keep `src/api/` thin — handlers parse, call a service, return a schema. Logic lives in `src/services/` and `src/core/`.

### TypeScript
- `strict: true`. Server Components by default; mark `'use client'` only when needed.
- Forms: `react-hook-form` + `zod`. API calls: `@tanstack/react-query`.
- Reuse shared DTO types from `packages/shared-types`.

## Knowledge base — `apps/api/knowledge/`

**This directory is the brain of the product. Treat it as production data.**

- `skills/<area>/SKILL.md` — narrative context for Claude when auditing
- `rules/*.yaml` — machine-readable rule definitions (zone weights, dosha triggers, remedy mappings)
- `data/*.json` — static taxonomies (deities, elements, colors)

Every rule must carry a `source` / citation field. Multi-traditional (North + South Indian Vastu) — never silently merge schools; tag rules with `tradition: north | south | universal`.

## Founder's directives

When in doubt: **simplicity > cleverness · proven libs > bleeding edge · readable > clever · typed > untyped · modular > tightly-coupled · ship fast > over-engineer.**

Code must read like another dev will join the team in 6 months — clean separation, clear docs, good tests.

## Don't

- Don't introduce new framework deps (e.g. Django, Express, Prisma, etc.) without explicit approval.
- Don't bypass Clerk for auth or Stripe for billing.
- Don't store plan images in Postgres — always R2.
- Don't write Vastu rules in code — they belong in `knowledge/rules/*.yaml`.
- Don't ship features without typed schemas at the API boundary.
